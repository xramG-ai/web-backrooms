# 웹 백룸 프로젝트 기획서 v2 (구현 검토 반영)

## v2 변경 요약

구현 관점에서 검토 후 다음을 수정·보강했다.

1. **좌표계 모순 해결** — 10km 셀 × 100 = 1,000km 월드는 float32 정밀도 한계(원점에서 수 km만 벗어나도 렌더/물리 지터 발생) 때문에 물리적으로 구현 불가. → 논리 좌표와 물리 좌표를 완전히 분리하고 **플로팅 오리진(월드 리베이스)** 방식으로 변경. 공간이 어차피 재배치되는 컨셉이므로 연속적인 물리 좌표 자체가 필요 없다는 점을 이용.
2. **층 높이 모순 해결** — Y셀도 10km이면 "층" 개념이 성립하지 않음. → 물리 층고는 3.2m, 논리 Y축은 층 인덱스로 정의.
3. **글리치 발생 경로 설계 추가** — Rapier 물리엔진을 쓰면 벽 관통이 우연히 일어나지 않음. 글리치를 콘텐츠로 쓰려면 **의도적으로 심어둔 균열**이 필요. → 절차 생성 규칙에 균열 배치 추가.
4. **재배치 판정 기준 구체화** — "시야 밖"의 구현 기준(프러스텀 + 거리 + 미관측 시간) 명시.
5. **시드/에포크 기반 절차 생성 구조 명시** — 재배치를 "에포크 증가 후 재생성"으로 통일해 상태 저장 최소화.
6. **MVP를 마일스톤 M0~M4로 세분화** — 검증 순서 명확화.
7. **기술 스택 세부 보강** — KinematicCharacterController, PointerLock, postprocessing, 인스턴싱, 성능 예산 추가.

세계관·연출 파트(1, 2, 6, 8, 9, 10, 14, 15)는 원안 유지.

## 1. 프로젝트 개요

### 프로젝트 형태
- 브라우저에서 실행되는 3D 웹 기반 백룸 탐색 프로젝트
- 일반적인 웹사이트보다는 사용자가 직접 공간에 진입해 걷고 탐색하는 형태
- 게임의 문법을 일부 사용하지만, 명확한 승리 조건이나 친절한 안내를 전제로 하지 않음
- 사용자가 공간의 규칙을 직접 해석해야 하는 구조

### 핵심 방향
이 프로젝트는 다음 세 가지 감각을 결합한다.

1. **백룸** — 노란 벽지, 반복되는 실내 공간, 낮은 가시거리, 익숙하지만 설명할 수 없는 구조
2. **영화 「큐브」의 공간 원리** — 전체 구조를 외우거나 지도화하기 어려움. 이동하는 동안 통로와 방의 연결 구조가 바뀜. 변화는 사용자가 확인하기 어려운 순간에 발생. 공간은 넓어서가 아니라 연결 관계가 계속 달라지기 때문에 무한하게 느껴짐
3. **글리치** — 벽 관통, 바닥 관통, 비정상 좌표 이동을 단순 버그가 아니라 시스템이 인식하는 세계관 요소로 취급. 허용 범위를 벗어나는 이동은 `글리치`로 판정하고, 이후 더 높은 난도의 공간 규칙이 적용될 수 있음

## 2. 세계관의 기본 개념

(원안 유지)

백룸은 현실에서 사라진 물건이나 사람이 떨어질 수 있는 공간이다. 현실의 표면이나 충돌 구조를 벗어난 대상이 백룸으로 이동하며, 프로젝트 내부에서는 이 현상을 `글리치`라고 부른다.

백룸은 내부 규칙과 좌표를 가진 하나의 시스템이다. 정상 범위에 있는 동안에는 노란 벽지 백룸의 형태를 유지하고, 범위를 벗어나면 같은 백룸이 점점 잘못 작동하는 방식으로 반응한다.

## 3. 전체 공간 구조

### 논리 좌표 (데이터 전용)
논리 좌표는 **순수 데이터 구조**이며, 물리/렌더 좌표와 어떤 직접적 대응도 갖지 않는다.

```text
X축: 1~100 (수평)
Z축: 1~100 (수평)
Y축: 1~100 (층 인덱스)
총 논리 셀: 100 × 100 × 100
```

각 셀이 저장하는 것: 시드, 층, 구역 타입(정상/이탈), 위험도, 변이 에포크.

### 시작 위치
```text
X = 50, Z = 50, Y = 40~60 무작위
```

- 사용자는 중앙 배치 사실과 층수를 알 수 없음 (UI 미표시)

### 층 구조
```text
정상 범위: 30~70층
하부 이탈: 1~29층
상부 이탈: 71~100층
```

경계값은 밸런스 조정 대상.

### [v2] 물리 좌표와의 관계 — 플로팅 오리진
**전제: float32 기반 Three.js/Rapier에서 물리 좌표가 원점에서 수 km 이상 벗어나면 정점 지터, 물리 불안정이 발생한다. 1,000km 연속 월드는 만들 수 없고, 만들 필요도 없다.**

해결 구조:

```text
- 플레이어의 물리 좌표는 항상 원점 ±500m 이내로 유지
- 재배치가 일어나는 은폐 순간(모서리 회전, 문 통과 등)에
  월드 전체를 원점 기준으로 리베이스(플레이어와 모든 청크를 동일 오프셋 이동)
- 논리 셀 전이는 물리 좌표가 아니라 "누적 보행 거리 + 진행 방향"으로 판정
```

이 구조의 부수 효과: 공간 재배치와 리베이스가 같은 순간에 일어나므로, 리베이스 자체가 컨셉("좌표는 존재하지 않는다")을 구현하는 수단이 된다.

## 4. 공간 단위와 규모

### [v2] 스케일 재정의
원안의 "논리 셀 1개 = 10,000m"는 **서사적 스케일**(사용자가 체감해야 하는 규모)로 유지하되, 구현 파라미터는 분리한다.

```text
서사적 셀 규모: 10,000m (사용자가 셀 하나의 전체 구조를 파악할 수 없어야 함)
논리 셀 전이 거리: 500~1,000m 누적 보행 시 인접 셀로 전이 (튜닝 파라미터)
물리 층고: 3.2m (일부 모듈은 2.4m 낮은 천장 / 6m 이상 홀 허용)
층 전이: 계단, 추락, 글리치를 통해서만 발생
```

시야가 10m로 제한되고 공간이 재배치되므로, 실제 걸은 거리와 논리 거리가 일치할 필요가 없다. 사용자는 차이를 검증할 수단이 없다.

### 사용자 이동
- WASD 이동, Shift 달리기, Space 점프, 마우스 시점 이동 (Pointer Lock)
- 보폭·속도는 일반 사람 기준: 걷기 ~1.5m/s, 달리기 ~4m/s

### 사용자 인식 거리
정면 유효 시야 약 10m. 단순 렌더 컷이 아니라 다음의 조합으로 구현:

- 지수 안개(FogExp2)
- 조명 감쇠, 형광등 밝기 제한
- 노란 벽지의 반복성
- 공간 음향의 왜곡
- 원거리 구조물의 명암 소실

## 5. 공간 재배치 규칙

### 핵심 원칙
통로와 방 연결 구조는 사용자가 인식할 수 없는 순간에 바뀐다. 직접적인 벽 이동 애니메이션은 최소화한다.

### [v2] 재배치 가능 판정 (구현 기준)
청크가 다음 세 조건을 **모두** 만족할 때만 재배치 대상:

```text
1. 카메라 프러스텀 밖 (또는 프러스텀 안이지만 안개 너머 > 25m)
2. 플레이어와의 거리 > 20m
3. 마지막 관측 후 경과 시간 > N초 (기본 5초, 난이도에 따라 감소)
```

절대 금지: 플레이어가 서 있는 청크, 진행 방향 정면 15m 이내 청크의 변경.

재배치 트리거(은폐 순간): 모서리 회전으로 후방 차폐, 문/좁은 통로 통과, 일정 걸음 수 누적, 조명 플리커 이벤트. 트리거 발생 시 대상 청크의 에포크를 증가시키고 다음 로드 때 재생성한다.

### 지도화 방지
- 동일한 벽지, 카펫, 형광등의 반복
- 랜드마크는 존재하되 위치와 주변 구조는 비고정
- 사용자가 놓은 물건도 재배치 대상 (MVP 이후)

> 랜드마크는 존재하지만, 좌표는 존재하지 않는다.

## 6. 정상 이동과 문 등장 규칙

(원안 유지)

정상 범위 내에서 글리치 없이 이동하면 1시간 이내 정면 10m 시야 안에 문이 등장한다.

[v2] 구현: DoorScheduler가 경과 시간 + 누적 이동 거리를 추적하고, 임계값 도달 시 **플레이어 진행 방향에 다음으로 생성될 청크**에 문 모듈을 주입한다. 등장 보장을 위해 잔여 시간이 10분 이하로 줄면 주입 확률을 100%까지 상승시킨다.

문은 출구가 아니다. 열리지 않을 수 있고, 뒤에 벽이 있을 수 있고, 다른 노란 방·복도·계단으로 이어질 수 있고, 아무 기능이 없을 수 있다. 의미를 설명하지 않는다.

## 7. 글리치 판정

### 정의
글리치는 사용자가 백룸 시스템의 정상 허용 좌표를 벗어났을 때 발생하는 공간 이탈 현상이다.

### [v2] 글리치는 저절로 일어나지 않는다 — 균열 설계
Rapier 같은 물리엔진을 정상적으로 쓰면 벽/바닥 관통은 사실상 발생하지 않는다. 글리치를 콘텐츠로 쓰려면 **의도적으로 심어둔 균열**이 필요하다.

절차 생성 시 낮은 확률로 배치하는 균열 유형:

```text
- 콜라이더가 누락된 벽 틈새 (시각적으로는 미세하게 어긋난 벽지 이음새)
- 달리기+점프로만 터널링 가능한 얇은 벽 구간
- 배선 노출 구역의 바닥 구멍 (카펫 텍스처는 정상으로 보임)
- 막다른 통로 끝의 "밀리는" 벽 (일정 시간 밀착 시 통과)
```

균열의 시각적 힌트는 아주 미세하게만 제공한다. 발견은 우연 또는 집요한 탐색의 결과여야 한다.

### GlitchDetector 판정 (매 프레임 또는 저주기 체크)
```text
- 하방 레이캐스트: 발밑에 바닥 콜라이더 없음 → 바닥 아래 판정
- 오버랩 쿼리: 플레이어 캡슐이 벽 볼륨 내부 → 벽 내부 판정
- Y 물리 좌표 범위 이탈 → 천장 위 / 낙하 판정
- 프레임당 이동량 > 임계값 → 비정상 속도 판정
- 논리 층 범위 이탈 (30 미만 / 70 초과) → 구역 이탈 판정
```

### 시스템 처리 원칙
즉시 원위치로 되돌리지 않는다.

```text
1. 비정상 이동 감지
2. 현재 좌표·진입 방식 기록
3. 글리치 이벤트 판정
4. 해당 위치에 맞는 이탈 난이도 적용
5. 안전망: 일정 시간 이상 무한 추락 시 화면 페이드 후
   이탈 구역 셀로 조용히 재배치 (원위치 복귀 아님)
```

> 버그는 허용하지만, 시스템이 인식하지 못하는 무제어 오류는 허용하지 않는다.

## 8. 난이도 구조

(원안 유지 — 인지 난이도와 심리 난이도 중심)

- **정상 구역(30~70층)**: 느린 구조 변화, 낮은 엔티티 빈도, 문 등장 보장
- **하부 이탈(≤29층)**: 습기, 저주파, 바닥 중심 이상, 형광등 어두움, 재배치 빈도 증가
- **상부 이탈(≥71층)**: 건조함, 고주파, 과도한 밝기, 천장 중심 이상, 탈색된 공간

난이도 상승은 수치 2배가 아니라 구조 변경 빈도, 엔티티 확률, 소리 왜곡, 문·통로 신뢰도, 조명 불안정, 방향 감각 교란, 복귀 경로 희소성의 강도 조절로 구현한다.

[v2] 구현: DifficultyDirector가 단일 위험도 스칼라(0~1)를 관리하고, 각 서브시스템(MutationSystem, EntityDirector, AudioDirector)이 이 값을 자기 파라미터로 매핑한다.

## 9. 엔티티 연출

(원안 유지)

- 발생 기준: 시간이 아니라 조작 누적(걷기, 달리기, 점프, 시점 회전, 문 상호작용 등)으로 판정
- 연출: 정면 5~8m 월드 공간에 배치된 빌보드 + 플래시 포스트프로세스 + 셔터/전기 파열음, 0.1~0.3초 노출 후 제거. 재확인 시 반드시 사라져 있어야 함. 발생 후 쿨다운 적용
- 결과 비고정: 아무 일 없음 / 통로 변경 / 발소리 / 문 등장·소멸 / 출현 확률 증가. 정상 구역에서는 심리적 경고, 이탈 구역에서만 실제 추적 이벤트 연결 가능

> 백룸 시스템 또는 엔티티가 사용자의 존재를 인식하고 있다.

## 10. 오디오 설계

(원안 유지)

- 기본: 형광등 전기음, 카펫 발소리, 원거리 구조 진동, 벽 내 배선음, 박자가 어긋난 유사 발소리
- 정상 구역: 일정한 형광등 소리, 낮은 잔향, 반복성 강조
- 이탈 구역: 방향-위치 불일치, 발소리 복제, 상부 고주파/하부 저주파 강화

[v2] 구현: Three.js `PositionalAudio`(내부적으로 Web Audio PannerNode)로 공간 음향 처리. 형광등 험은 오실레이터+노이즈로 절차 생성해 루프 티가 나지 않게 한다. 이탈 구역의 "방향 불일치"는 패너 위치를 실제 음원과 다르게 배치하는 방식으로 구현.

### 렌더링 원칙
`100³` 전체를 메시로 만들지 않는다. 논리 공간과 렌더 공간을 분리한다.

```text
논리 공간: 100×100×100, 셀별 시드·층·규칙·위험도·에포크만 저장 (수 MB 이하)
렌더 공간: 플레이어 주변 반경 ~40m의 청크만 생성, 밖은 해제/재사용
```

권장 파라미터:

```text
유효 시야: 10m
안개 완전 차폐: ~25m
렌더 반경: 40m (안개 덕에 짧아도 됨 — 원안의 80m는 불필요)
청크 크기: 20m × 20m (층고 3.2m)
```

### 성능 예산 (60fps 기준)
```text
- 드로우콜 < 300 (벽/바닥/천장/형광등/기둥/문틀은 InstancedMesh)
- 실시간 라이트 최소화: 형광등은 emissive 텍스처 + 소수의 실제 라이트만
- 청크 생성/해제는 프레임당 1개 이하로 분산 (히치 방지)
- 물리 콜라이더는 청크당 병합된 소수의 cuboid로
```

### 시스템 구조

```text
WorldManager
├─ LogicalGrid        // 100³ 셀 데이터, 시드·에포크 관리
├─ ChunkManager       // 로드/언로드, 플로팅 오리진 리베이스
├─ RoomGenerator      // 시드 → 모듈 조합 → 청크 지오메트리
├─ CorridorConnector  // 청크 경계 통로 정합성 보장
├─ VisibilityTracker  // 프러스텀·거리·최종 관측 시각 추적
├─ MutationSystem     // 재배치 트리거 → 에포크 증가
├─ DoorScheduler      // 시간+거리 누적 → 문 주입
├─ GlitchDetector     // 레이캐스트·오버랩·범위 체크 (물리와 분리)
├─ DifficultyDirector // 위험도 스칼라 관리
├─ EntityDirector     // 조작 누적 → 플래시 이벤트
└─ AudioDirector      // 구역별 음향 파라미터
```

## 12. 절차 생성 구조

### [v2] 시드와 에포크
재배치를 "상태 변경"이 아니라 "재생성"으로 통일한다.

```text
chunkSeed = hash(cellX, cellY, cellZ, chunkId, epoch)

- 같은 에포크로 재방문 → 같은 구조 (일관성)
- MutationSystem이 에포크 증가 → 다음 로드 시 다른 구조 (재배치)
- 저장해야 하는 상태는 셀별 에포크 정수 하나뿐
```

경계 정합성: 인접 청크의 통로 연결점은 경계 좌표의 해시로 결정해 양쪽이 독립적으로 생성해도 항상 맞물리게 한다. 단, 에포크가 다른 청크끼리는 의도적으로 어긋날 수 있다 — 이 어긋남(막힌 통로, 반쪽 문)도 세계관 요소로 쓴다.

### 공간 모듈
직선 복도, L자 복도, T자 분기, 넓은 사무 공간, 기둥 밀집 구역, 낮은 천장 구역, 막힌 통로, 문이 있는 벽, 계단 구역, 배선 노출 구역, 젖은 카펫 구역, 형광등 꺼진 구역, **[v2] 균열 모듈(7절 참고)**

### 구조 변경 단계
- 소규모: 출입구 위치, 벽 추가/제거, 조명 상태, 문 생성/제거
- 중규모: 복도 묶음 연결, 계단 위치, 분기 방향, 지나온 구역 재배치
- 대규모: 논리 셀 연결, 층 연결, 위험도, 엔티티 규칙, 오디오 환경 변경

## 14. 제작 원칙

(원안 유지)

**사용자에게 설명하지 않는 것**: 현재 층, 좌표, 정상 범위, 글리치 판정 기준, 문 등장 시간, 엔티티 카운터, 재배치 규칙, 맵 크기, 중앙 시작 사실

**시스템이 알아야 하는 것**: 논리 좌표, 층, 정상/이탈 상태, 이동 거리, 조작 누적, 구조 변경 횟수, 문 생성 여부, 플래시 기록, 글리치 위치·방식, 렌더 중인 청크, 시야 내 구조물

**핵심 태도**: 안내하지 않는다. 이해를 돕지 않는다. 버그처럼 보이는 현상을 세계관으로 쓴다. 단, 브라우저 정지와 진행 불가 상태는 방지한다. 공간은 불친절하지만 시스템은 무너지지 않는다.

## 15. 핵심 요약

100³ 논리 좌표를 가진 노란 벽지 백룸. 사용자는 수평 중앙, 40~60층에서 시작하고 정면 약 10m만 인식한다. 공간은 시야 밖에서 연결 구조를 바꾸며 지도화할 수 없다. 30~70층이 정상 범위, 벗어나면 이탈 구역. 정상 이동 시 1시간 내 문이 등장하지만 문은 출구가 아니다. 조작 누적에 따라 엔티티 얼굴이 플래시처럼 나타난다.

구현의 핵심은 세 가지 분리다: **논리 좌표와 물리 좌표의 분리**(플로팅 오리진), **물리 충돌과 글리치 감지의 분리**, **상태 저장과 절차 생성의 분리**(시드+에포크).

> 공간은 존재하지만 이해할 수 없고, 길은 이어지지만 어디로 가는지 알 수 없으며, 시스템은 사용자를 보고 있지만 사용자는 시스템을 볼 수 없다.
